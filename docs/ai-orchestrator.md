# AI Orchestrator Guide

> AI 에이전트와 협업하는 개발자의 역할 정의

---

## Overview

| 항목 | 설명 |
|------|------|
| 목적 | AI 에이전트를 효과적으로 통제하고 협업하기 위한 오케스트레이터 역할 정의 |
| 대상 | AI 기반 개발 워크플로우를 도입하려는 개발자/팀 |
| 핵심 원칙 | 명확한 규칙 > 모호한 명령 (No Vibe Coding) |

## 1. Constraints & Rules (제약 조건 및 규칙)

### 목표
AI가 따라야 할 명확한 규칙과 제약 조건을 설정한다.

### 실행 항목

| 항목 | 설명 | 예시 |
|------|------|------|
| 규칙 문서화 | 프로젝트 규칙을 파일로 명문화 | `.cursorrules`, `CLAUDE.md`, `AGENTS.md` |
| 아키텍처 패턴 | 허용/금지 패턴 명시 | "Repository 패턴 사용", "직접 DB 접근 금지" |
| 네이밍 컨벤션 | 일관된 명명 규칙 | `camelCase` for variables, `PascalCase` for classes |
| 절차서(Procedure) | 반복 작업의 표준화된 절차 | PR 생성 절차, 테스트 작성 절차 |

### 체크리스트
- [ ] 프로젝트 루트에 AI 규칙 파일 존재 여부
- [ ] 금지 사항 목록 명시 여부
- [ ] 코드 스타일 가이드 문서화 여부

## 2. Verification System (검증 시스템)

### 목표
AI 결과물의 품질을 자동화된 시스템으로 보장한다.

### 실행 항목

| 항목 | 설명 | 도구 예시 |
|------|------|----------|
| CI/CD 가드레일 | 테스트/린트 실패 시 머지 차단 | GitHub Actions, GitLab CI |
| 테스트 자동화 | AI 코드 검증의 핵심 수단 | Jest, Pytest, Vitest |
| Self-healing | 테스트 실패 시 AI 자동 수정 루프 | CI + AI Agent 연동 |

### 검증 파이프라인

```
[AI 코드 작성]
    ↓
[자동 테스트 실행]
    ↓
[실패?] → [AI 로그 분석] → [자동 수정] → [재커밋]
    ↓
[성공] → [리뷰 요청]
```

### 체크리스트
- [ ] 테스트 커버리지 기준 설정
- [ ] 린트/포맷 검사 자동화
- [ ] 테스트 없는 코드 머지 차단 규칙

## 3. Strategic Rebuild (전략적 재구축)

### 목표
기술 부채 대신, AI를 활용한 시스템 재구축을 표준화한다.

### 실행 항목

| 전략 | 설명 | 적용 시점 |
|------|------|----------|
| 주기적 시스템 환생 | 기존 로직 분석 후 최신 스택으로 재구축 | 기술 부채 누적 시 |
| 프로토타입 폐기 | 프로토타입 → 즉시 프로덕션 레벨 재구축 | MVP 검증 완료 후 |
| 마이그레이션 회피 | 점진적 수정 대신 전체 재작성 | 레거시 현대화 시 |

### 의사결정 기준

```
IF 수정 비용 > 재구축 비용 THEN 재구축
IF 기술 부채 > 임계값 THEN 재구축 검토
IF 프로토타입 검증 완료 THEN 폐기 후 재구축
```

## 4. Context Management (맥락 관리)

### 목표
AI가 시스템 전체를 이해하고 작업할 수 있는 환경을 구축한다.

### 실행 항목

| 항목 | 설명 | 이점 |
|------|------|------|
| 모노레포 운용 | 모든 서비스를 단일 저장소에 | 의존성/API 충돌 파악 용이 |
| 요약 기반 리뷰 | AI의 변경사항 요약서 검토 | 리뷰 효율성 향상 |
| 핵심 로직 승인 | 중요 로직만 개발자가 최종 승인 | 리스크 관리 |

### 맥락 제공 구조

```
project/
├── CLAUDE.md          # AI 규칙 및 컨텍스트
├── docs/
│   ├── architecture.md    # 시스템 아키텍처
│   ├── api-spec.md        # API 명세
│   └── conventions.md     # 코딩 컨벤션
├── .github/
│   └── workflows/         # CI/CD 파이프라인
└── src/                   # 소스 코드
```

## Quick Reference

### 오케스트레이터의 4대 역할

| # | 역할 | 핵심 질문 |
|---|------|----------|
| 1 | 제약 조건 정의 | "AI가 따라야 할 규칙이 명문화되어 있는가?" |
| 2 | 검증 시스템 구축 | "AI 결과물을 자동으로 검증할 수 있는가?" |
| 3 | 전략적 재구축 | "수정보다 재구축이 효율적인가?" |
| 4 | 맥락 관리 | "AI가 전체 시스템을 이해할 수 있는가?" |

### 핵심 원칙

```
1. 모호한 명령 금지 → 명확한 규칙 문서화
2. 직접 코드 리뷰 대신 → 자동화된 검증
3. 점진적 수정 대신 → 전략적 재구축
4. 분산된 정보 대신 → 통합된 맥락 제공
```
